## 📋 Description

A pure service function that maps a player's current level (integer 1–100) to a Sudoku board configuration. Returns `board_size` (4, 6, or 9), `hidden_count` (number of cells to hide), `error_limit` (maximum wrong submissions allowed before the session ends), and `score_multiplier` (float multiplier applied to the base score formula). The exact level-to-config mapping is defined in a separate level design specification. This function has no API endpoint and is called internally by SF01 during session initialization.

Only three board sizes are supported: **4×4** (block shape 2×2), **6×6** (block shape 2×3), **9×9** (block shape 3×3).

## 🗏️ Business Logic (3 Steps)

**Precondition**: Called internally by SF01 after `player_level` has been loaded from `user_profiles`. This function does not perform authentication or database access.

1. **Validate Player Level** - Assert `1 <= player_level <= 100`; raise `ValueError("INVALID_LEVEL")` (translated to HTTP 400 at the service boundary in SF01) if out of range
2. **Look Up Config Table** - Match `player_level` against the defined level range table to find the corresponding config entry; the table is a static mapping (dict or list of range buckets) defined as a module-level constant in `sudoku_service.py`
3. **Return SudokuConfig** - Construct and return a `SudokuConfig` instance with `board_size`, `hidden_count`, `error_limit`, `score_multiplier` from the matched entry

### Architecture Overview

| Component | Purpose | Details |
|-----------|---------|---------|
| **Service Function** | Pure mapping logic | `resolve_sudoku_config(player_level)` — no DB access, no side effects |
| **Config Table** | Level-to-config mapping | `SUDOKU_LEVEL_CONFIG`: module-level constant (list of dicts or `dict[range, SudokuConfig]`) |
| **Pydantic Schema / Dataclass** | Return type | `SudokuConfig(board_size, hidden_count, error_limit, score_multiplier)` in `app/schemas/sudoku.py` |
| **Error Handling** | Input validation | `ValueError` for out-of-range level; translated to HTTP 400 at service boundary |

### Implementation Highlights

⬜ **SudokuConfig type**: Define `SudokuConfig` as a Pydantic BaseModel or Python dataclass in `app/schemas/sudoku.py` with fields `board_size: int`, `hidden_count: int`, `error_limit: int`, `score_multiplier: float`  
⬜ **SUDOKU_LEVEL_CONFIG constant**: Static level range table defined at module level in `sudoku_service.py`; covers all levels 1–100 without gaps or overlaps  
⬜ **Level validation**: Guard clause at function entry; raises `ValueError` for level < 1 or level > 100  
⬜ **Range lookup**: Iterate or binary-search the config table to find the matching level bucket  
⬜ **Pure function**: No database calls, no async, no side effects — straightforward deterministic mapping  
⬜ **Board size constraint**: Config table only maps to `board_size` values of `4`, `6`, or `9`; no other values are valid  

### Future Enhancements

- Load level config from database or external config file to allow hot-reload without code deployment
- Support fractional or decimal `player_level` input if the level system gains sub-levels
- Expose a read-only admin endpoint to inspect the full level config table

## 📊 Security Considerations

| Area | Implementation |
|------|----------------|
| **Input Validation** | `player_level` range check [1, 100] prevents unexpected config lookups or index-out-of-bounds errors |
| **No Database Access** | Function is pure; no risk of SQL injection or DB side-effects |
| **Config Integrity** | `SUDOKU_LEVEL_CONFIG` is a read-only module constant; cannot be modified at runtime by client input |
| **No External Input** | `player_level` is sourced from the server-side `user_profiles` table (set by SF01), not from client-supplied request data |
